# Git 커밋 메시지 컨벤션

## 1. 기본 개념

- 커밋 메시지는 다음 형식을 사용한다.
  - `type: subject`
  - (선택) 빈 줄
  - (선택) body
  - (선택) 빈 줄
  - (선택) footer
- `type`은 **영어**를 사용한다.
- `subject`, `body`, `footer` 내용은 **한국어**로 작성한다.

---

## 2. 커밋 타입

### 2-1. 자주 사용하는 기본 타입

- **feat**  
  - 새로운 기능 추가

- **fix**  
  - 버그 수정

- **refactor**  
  - 동작 변화 없이 코드 구조/로직 개선

- **chore**  
  - 빌드 설정, 패키지 관리, CI/CD, 환경 설정 등
  - 애플리케이션 기능과 직접 관련 없는 작업

- **test**  
  - 테스트 코드 추가/수정

### 2-2. 상황에 따라 사용하는 보조 타입

- **comment**  
  - 코드 **주석만** 추가/수정

- **rename**  
  - 파일/폴더/변수 이름 변경 (동작 변화 없음)

- **remove**  
  - 파일/코드 삭제가 주요 목적일 때

- **design**  
  - CSS, 레이아웃, 색상, 폰트 등 UI/UX 스타일 수정  
  - 화면 작업이 있을 때만 사용 (현재는 사용 빈도 낮음)

---

## 3. 제목(subject) 규칙

- 50자 이내로 간결하게 작성한다.
- `type` 뒤 **첫 글자는 소문자**로 시작한다.
- 문장 끝에 마침표(`.`)를 붙이지 않는다.
- 커밋 메시지만 보고도 “무엇이 바뀌었는지” 이해할 수 있게 한국어로 작성한다.  
  - 예) `feat: 회원가입 폼 유효성 검사 추가`  
  - 예) `fix: 로그인 실패 시 에러 메시지 노출 문제 수정`

---

## 4. 본문(body) / 옵션(footer)

### 4-1. body

- 제목만으로 설명이 부족할 때만 작성한다.
- “**무엇을 변경했는지** + **왜 변경했는지**”를 한국어로 서술한다.
- 제목과 body 사이에는 **빈 줄 1줄**을 둔다.
- 필요할 때만 작성하며, 필수는 아니다.

### 4-2. footer

- 관련 이슈 번호, 참고한 커밋 ID를 명시할 때 사용한다.
- body가 있을 경우, body와 footer 사이에 **빈 줄 1줄**을 둔다.
- 예시
  - `Refs: #123`
  - `Fixes: #45`
  - `See: abcdef1`

> 기본적으로는 `type: subject`만 사용하고,  
> 필요한 경우에만 body/footer를 추가한다.

---

## 5. 헷갈릴 때 판단 기준

- **리팩터링하면서 파일 이름도 같이 바꿨다**
  - 전체 구조/로직 개선이 목적 → `refactor`
  - 이름만 바꾼 커밋 → `rename`

- **주석도 고치고 코드도 약간 고쳤다**
  - 실제 로직이 바뀜 → `feat` / `fix` / `refactor` 중 선택
  - 진짜 주석만 건드림 → `comment`

- **사용하지 않는 파일/코드를 지웠다**
  - 삭제 자체가 핵심 목적 → `remove`
  - 리팩터링 과정에서 자연스럽게 사라짐 → 전체 맥락에 따라 `refactor`

---

## 6. 공통 원칙

- **의미 있는 단위로 커밋**
  - 한 커밋에 너무 많은 종류의 변경을 섞지 않는다.
  - 기능/수정 단위로 커밋을 잘게 나눈다.

- **명확하게 작성**
  - 커밋 메시지만 보고도 변경 요지를 파악할 수 있어야 한다.
  - “수정”, “업데이트”처럼 추상적인 단어만 쓰지 않는다.

- **협업 정보 포함**
  - 영향받는 모듈/파일, 관련 이슈 번호 등이 있다면 가능하면 메시지에 포함한다.
